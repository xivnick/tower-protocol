# AGENTS.md

- 작업은 기본적으로 제안, 구현, 머지의 세 단계로 진행한다.
- 제안 단계에서는 변경 방향과 구현 계획을 짧게 제시하고 승인받는다.
- 사용자가 "구현", "진행", "작업해"처럼 구현 착수를 명시하면 작업 규모와 무관하게 제안 승인 없이 즉시 구현한다. 이 규칙은 제안 단계보다 우선한다.
- 큰 작업은 첫 번째 동작 가능한 수직 단위로 분할한다. 각 단위는 같은 작업 흐름에서 코드 변경, 검증, 커밋, 푸시까지 완료한 뒤 다음 단위로 진행한다.
- 명시적 구현 지시 후에는 실제 파일 변경, 검증, 커밋 중 하나도 없이 계획이나 상태 보고만으로 응답을 종료하지 않는다.
- "작업 중"이라고 보고할 때는 변경한 파일 또는 실행 중인 검증과 현재 완료 단위를 함께 제시한다.
- 새 작업은 최신 `main`에서 `feature/<작업명>` 브랜치를 생성해 시작하고, `main`에는 직접 작업하거나 커밋하지 않는다.
- 구현 단계에서는 피처 브랜치에서 작업하고, 검증 후 커밋과 푸시까지 완료한다.
- 머지 단계에서는 사용자 확인 후 PR을 생성하고, 머지와 원격·로컬 브랜치 삭제까지 처리한다.
- 구현 단계에서는 앱 버전, `version.json`, 패치노트 migration을 변경하지 않는다.
- 버전 변경과 패치노트 작성·적용은 머지 단계에서 사용자 확인 후 처리한다.

- 기능은 작고 검증 가능한 단위로 구현하고, 현재 기능이 요구하지 않는 시스템은 미리 설계하지 않는다.
- 기존 프로젝트 패턴을 우선하고, 아키텍처 경계를 깨지 않는다.
- Supabase는 서버 권위 백엔드로 취급한다.
- DB, 인프라, 배포, 원격 서비스 변경은 아래 Supabase 마이그레이션 예외를 제외하고 먼저 확인한다.
- 기존 버전과 호환되고 데이터 손실·변환 위험이 없는 Supabase 마이그레이션은 구현 단계에서 원격 DB에도 적용한다. 데이터 손실, 변환 위험, 권한·인프라·배포 변경이 있는 경우에만 먼저 확인한다.
- 패치노트만을 위한 Supabase 마이그레이션은 릴리스 작업으로 취급하고 머지 단계 전에는 원격 DB에 적용하지 않는다.
- 원격 Supabase DB 작업 시 `app/.env.local`의 `SUPABASE_DB_PASSWORD`를 불러와 사용한다. 값은 출력, 문서화, 커밋하지 않는다.
- 프론트 변경 후에는 빌드를 기본 검증으로 삼고, 브라우저 확인은 시각·상호작용 검증이 필요할 때만 수행한다.
- 터미널/CLI 판타지 RPG 스타일과 짧고 기능적인 앱 문구를 유지한다.
