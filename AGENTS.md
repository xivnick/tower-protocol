# AGENTS.md

- 작업은 기본적으로 제안, 구현, 머지의 세 단계로 진행한다.
- 제안 단계에서는 변경 방향과 구현 계획을 짧게 제시하고 승인받는다.
- 사용자가 바로 구현을 지시한 작고 명확한 작업은 제안 단계를 생략할 수 있다.
- 구현 단계에서는 피처 브랜치에서 작업하고, 검증 후 커밋과 푸시까지 완료한다.
- 머지 단계에서는 사용자 확인 후 PR을 생성하고, 머지와 원격·로컬 브랜치 삭제까지 처리한다.

- 기능은 작고 검증 가능한 단위로 구현하고, 현재 기능이 요구하지 않는 시스템은 미리 설계하지 않는다.
- 기존 프로젝트 패턴을 우선하고, 아키텍처 경계를 깨지 않는다.
- Supabase는 서버 권위 백엔드로 취급한다.
- DB, 인프라, 배포, 원격 서비스 변경 전에는 먼저 확인한다.
- 프론트 변경 후에는 빌드를 기본 검증으로 삼고, 브라우저 확인은 시각·상호작용 검증이 필요할 때만 수행한다.
- 터미널/CLI 판타지 RPG 스타일과 짧고 기능적인 앱 문구를 유지한다.
