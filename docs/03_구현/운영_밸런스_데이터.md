# 운영 밸런스 데이터

배포 후 몬스터, 정수, 탑, 드랍 수치를 어드민에서 조정하기 위한 데이터 운영 기준.

## 기본 방향

- 밸런스 데이터는 코드에 고정하지 않고 버전이 있는 데이터로 관리한다.
- 어드민에서는 정수, 몬스터, 탑 층, 드랍 테이블 같은 개별 항목을 수정한다.
- 배포 시에는 개별 항목을 모아 하나의 밸런스 스냅샷을 만든다.
- 게임 서버는 활성화된 스냅샷을 메모리에 로드하고, 새 전투부터 해당 버전을 사용한다.

```txt
어드민 편집 데이터: 개별 entity
배포 결과물: compiled balance snapshot
게임 런타임: active snapshot cache
```

## ID 규칙

내부 참조 ID에는 표시명을 넣지 않는다.

```txt
mon_001 ~ mon_100
ess_001 ~ ess_100
area_001 ~ area_020
floor_001 ~ floor_100
```

표시명은 언제든 바뀔 수 있으므로 ID와 분리한다.

```json
{
  "id": "mon_005",
  "slug": "forest-guardian-stag",
  "name": {
    "ko": "숲지기 큰사슴"
  },
  "areaId": "area_001",
  "essenceId": "ess_005",
  "towerRole": "boss"
}
```

```json
{
  "id": "ess_005",
  "slug": "forest-guardian-charge",
  "name": {
    "ko": "숲지기 큰사슴의 돌진"
  },
  "sourceMonsterId": "mon_005"
}
```

역할:

| 필드 | 용도 |
|---|---|
| `id` | 저장, 참조, 유저 보유 데이터에 쓰는 불변 내부 키 |
| `slug` | 어드민 검색, 로그, URL 등에 쓰는 사람이 읽기 쉬운 보조 키 |
| `name` | 화면에 표시하는 이름 |

## 정수 효과 데이터

정수 효과는 문장 파싱에 의존하지 않고 구조화 데이터로 저장한다.
현재 seed 데이터는 1~5등급을 모두 가진다.
4등급은 완전 문장으로 확정된 효과를 기준으로 구조화하고, 1/2/3/5등급은 100층 정수 얼라인 문서의 수치 방향을 기준으로 생성한다.
복잡한 트리거, 스택, 상태 효과는 자동 파싱 후 수동 override를 적용한다.

```txt
data/balance/manual-effect-overrides.json
```

```json
{
  "id": "ess_005",
  "type": "active",
  "cooldown": 6,
  "grades": {
    "4": {
      "effects": [
        {
          "type": "damage",
          "target": "enemy",
          "damageType": "physical",
          "scalingStat": "physicalAttack",
          "coefficient": 2.4
        },
        {
          "type": "buff",
          "target": "self",
          "stat": "physicalAttack",
          "percent": 0.16,
          "duration": 3
        }
      ]
    }
  }
}
```

UI 설명문은 구조화 데이터를 기반으로 자동 생성한다.

```txt
쿨타임 6초
대상에게 물리 공격력 240% 피해를 준다.
3초 동안 자신의 물리 공격력이 16% 증가한다.
```

기획 문구를 다듬어야 하는 경우 `summary`나 `descriptionOverride`를 둘 수 있다.
단, 전투 계산과 시뮬레이터는 항상 `effects`만 사용한다.

## 저장 구조

Supabase PostgreSQL을 기준으로 SQL 테이블과 JSONB를 함께 사용한다.

SQL 컬럼은 버전, 상태, 작성자, 배포일 같은 관리 정보에 사용한다.
밸런스 본문은 JSONB로 저장한다.

```txt
balance_versions
- id
- version
- status: draft / published / archived
- is_active
- base_version_id
- compiled_jsonb
- changelog_jsonb
- created_by
- published_by
- created_at
- updated_at
- published_at
```

```txt
balance_entities
- id
- version_id
- entity_type: rule / area / monster / essence / tower_floor / drop_table
- entity_id
- data_jsonb
- updated_by
- updated_at
```

어드민은 `balance_entities`를 기준으로 개별 항목을 보여주고 수정한다.
배포 시 서버는 같은 `version_id`의 모든 entity를 모아 `compiled_jsonb` 스냅샷을 만든다.
구체적인 DB 마이그레이션은 [[Supabase_스키마]]와 `supabase/migrations/001_balance_admin_schema.sql`에 둔다.

## 개별 entity가 필요한 이유

어드민의 작업 단위는 밸런스 전체가 아니라 정수 하나, 몬스터 하나, 탑 층 하나다.

개별 entity 방식의 장점:

- 정수/몬스터/층 목록 화면을 만들기 쉽다.
- 타입, 태그, 쿨타임, 구간 같은 기준으로 검색하고 필터링하기 쉽다.
- 여러 관리자가 서로 다른 항목을 수정할 때 충돌이 줄어든다.
- 개별 항목의 변경 이력과 검증 메시지를 보여주기 쉽다.
- 정수 수정 권한, 몬스터 수정 권한, 배포 권한을 분리하기 쉽다.

## 스냅샷이 필요한 이유

스냅샷은 특정 시점의 밸런스 전체를 확정된 상태로 고정한다.

스냅샷의 용도:

- 배포 버전 고정
- 즉시 롤백
- 전투 재현
- 시뮬레이션 결과 기준 고정
- 서버/클라이언트 캐싱
- 참조 관계가 맞는 데이터 묶음을 원자적으로 배포

```txt
balance 1.0.3 =
rules
areas
monsters
essences
towerFloors
dropTables
```

published 스냅샷은 수정하지 않는다.
수정이 필요하면 active 버전을 복사해 draft를 만들고, 새 버전으로 publish한다.

## 어드민 반영 흐름

```txt
1. active 버전을 복사해 draft 생성
2. 어드민에서 개별 entity 수정
3. 스키마 검증
4. 참조 ID 검증
5. 시뮬레이터 검증
6. publish
7. compiled snapshot 생성
8. active version 변경
9. 게임 서버가 새 active snapshot 로드
10. 새 전투부터 새 밸런스 적용
```

어드민에서 값을 수정해도 즉시 유저 전투에 반영되지 않는다.
검증과 publish를 거쳐 active version이 바뀔 때 반영된다.

진행 중인 전투는 시작 시점의 `balanceVersion`으로 끝까지 처리한다.
새 밸런스는 새로 시작되는 전투부터 적용한다.

## 어드민 화면

권장 메뉴:

```txt
Balance Admin
- Versions
- Essences
- Monsters
- Tower Floors
- Drop Tables
- Rules
- Simulations
- Publish
```

정수 편집 화면 예:

```txt
정수 ID: ess_005
이름: 숲지기 큰사슴의 돌진
타입: active
쿨타임: 6초

4등급 효과
- damage / physical / physicalAttack 240%
- buff / self / physicalAttack +16% / 3초
```

효과는 텍스트 에디터가 아니라 구조화 폼으로 수정한다.
어드민은 저장 전에 설명문 미리보기와 검증 결과를 보여준다.

## 검증 기준

publish 전 필수 검증:

- `schemas/balance-entity.schema.json` 통과
- `schemas/compiled-balance.schema.json` 통과
- 모든 `id` 중복 없음
- 모든 참조 ID 존재
- 정수 등급별 필수 필드 존재
- 효과 타입별 필수 파라미터 존재
- 수치 범위 검증
- 1~100층 시뮬레이션 실행 가능
- `NaN`, `undefined`, 파싱 실패 없음
