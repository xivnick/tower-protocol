# 개발 운영 원칙

## 브랜치와 배포

- `main` 브랜치는 production 배포 기준으로 사용한다.
- 기능 작업은 `feature/*` 브랜치에서 진행한다.
- Vercel preview를 확인한 뒤 `main`에 merge한다.
- 작업 단위가 끝나면 하나의 의미 있는 커밋으로 남긴다.
- PR merge 후 완료된 remote/local feature 브랜치는 삭제한다.

## 환경

- 환경은 `local`, `preview`, `production`으로 구분한다.
- 초기에는 Supabase 프로젝트와 DB를 하나만 사용한다.
- Preview 배포도 같은 Supabase DB를 바라볼 수 있으므로, 원격 DB 변경은 별도 확인 후 진행한다.
- Supabase 프로젝트를 환경별로 나누는 것은 비용과 운영 필요성이 생겼을 때 검토한다.

## 릴리스와 패치노트

- 버전은 `MAJOR.MINOR.PATCH` 형식을 따른다.
- 버전의 기준값은 `app/package.json`의 `version`이다.
- 앱의 최신·최소 지원 버전 조회는 Supabase `patch_notes` 기반 `get_app_version()`을 우선 사용한다.
- `app/public/version.json`은 업데이트 조회 fallback으로 유지한다.
- 기능 구현 단계에서는 앱 버전을 올리지 않는다.
- 기능 구현 단계에서는 `app/public/version.json`을 변경하지 않는다.
- 기능 구현 단계에서는 패치노트 migration을 만들거나 원격 DB에 적용하지 않는다.
- 버전 변경, `version.json` 갱신, 패치노트 migration 작성·적용은 머지 단계에서 사용자 확인 후 처리한다.
- 작은 수정과 버그 수정은 `PATCH`를 올린다.
- 기능 단위 변경은 `MINOR`를 올린다.
- 호환성을 크게 깨는 변경이나 정식 공개 기준이 생기면 `MAJOR` 변경을 검토한다.
- 사용자 공개 변경은 기능 구현과 같은 릴리스 버전을 사용한다. 패치노트만을 위한 독립 버전은 만들지 않는다.

### 릴리스 파일

- 새 릴리스 PR은 아래 파일을 함께 변경한다.
  - `app/package.json`
  - `app/package-lock.json`
  - `app/public/version.json`
  - 새 `supabase/migrations/*_add_*_patch_note.sql`
- 패치노트 migration의 `patch_notes.version`은 `app/package.json.version`과 같아야 한다.
- `app/public/version.json`의 `latest`는 `app/package.json.version`과 같아야 한다.
- `minimum`은 기존 클라이언트를 강제로 새로고침해야 할 때만 `latest`와 같은 버전으로 올린다. 일반적인 호환 기능 추가에서는 기존값을 유지한다.
- 하나의 PR에 여러 기능이 있어도 사용자 공개 버전은 하나만 사용한다.

### 패치노트 문구

- 패치노트는 사용자에게 보이는 변화만 기록한다.
- 내부 구현 변경은 사용자가 체감하는 변화가 있을 때만 적는다.
- 문장은 짧고 기능적으로 쓴다.
- `tower://` 테마를 약하게 반영하되 과한 농담이나 장황한 설명은 피한다.
- 각 버전은 1~3개 항목으로 시작하고, 필요할 때만 늘린다.
- `patch_notes`에는 버전·출시일·제목을, `patch_note_items`에는 사용자 공개 항목을 기록한다.

### 릴리스 검증

- `npm run build`를 실행한다.
- 적용 후 `npm run db:list`로 로컬과 원격 마이그레이션 이력이 일치하는지 확인한다.
- Preview에서 `/version.json`이 `app/package.json.version`과 같은 `latest`를 반환하는지 확인한 뒤 merge한다.

## Vercel 배포

- Vercel 프로젝트의 Root Directory는 `app`으로 둔다.
- Build Command는 `npm run build`로 둔다.
- Output Directory는 `dist`로 둔다.
- SPA 직접 접근과 새로고침을 위해 `app/vercel.json`의 rewrite를 유지한다.
- Preview URL에서 `/`, `/reset-password`, `/version.json`을 확인한 뒤 merge한다.

## Supabase Auth Redirect

- 비밀번호 재설정 redirect 경로는 `/reset-password`를 사용한다.
- Supabase Auth Redirect URLs에는 local, preview, production URL을 등록한다.
- 로컬 개발 주소 예: `http://127.0.0.1:5173/**`.
- Vercel preview 테스트가 필요하면 preview URL 또는 `https://*.vercel.app/**`를 등록한다.
- production 도메인을 연결하면 `https://도메인/**`과 필요한 `www` 도메인을 함께 등록한다.

## 인증 상태

- 인증 화면은 명시적인 상태 모델을 따른다.
- 예: `checking`, `signed-out`, `profile-required`, `signed-in`, `reset-request`, `reset-update`.
- 로그인, 회원가입, 비밀번호 재설정, 프로필 생성 흐름을 한 상태에 섞지 않는다.

## 마이그레이션

- 마이그레이션 번호를 재사용하지 않는다.
- 이미 원격에 적용된 migration은 수정하지 않는다.
- 변경은 새 migration으로 추가한다.
- 원격 migration history와 로컬 파일 목록이 맞는지 확인한다.
- 기존 설정이나 데이터 구조를 크게 바꾸는 migration은 서버 점검 작업으로 취급한다.
- destructive migration은 별도 확인 후 진행한다.

## 서버 권위

- 클라이언트는 입력과 화면 표시를 담당한다.
- 전투 결과, 보상, 재화, 인벤토리 변경, 랭킹 기록은 서버가 확정한다.
- 서버 권위 로직은 필요한 시점에 Supabase Edge Functions로 추가한다.
