# 개발 운영 원칙

## 브랜치와 배포

- `main` 브랜치는 production 배포 기준으로 사용한다.
- 기능 작업은 `feature/*` 브랜치에서 진행한다.
- Vercel preview를 확인한 뒤 `main`에 merge한다.
- 작업 단위가 끝나면 하나의 의미 있는 커밋으로 남긴다.

## 환경

- 환경은 `local`, `preview`, `production`으로 구분한다.
- 초기에는 Supabase 프로젝트와 DB를 하나만 사용한다.
- Preview 배포도 같은 Supabase DB를 바라볼 수 있으므로, 원격 DB 변경은 별도 확인 후 진행한다.
- Supabase 프로젝트를 환경별로 나누는 것은 비용과 운영 필요성이 생겼을 때 검토한다.

## 버전 관리

- 버전은 `MAJOR.MINOR.PATCH` 형식을 따른다.
- 초기 개발은 `0.1.0`에서 시작한다.
- 작은 수정과 버그 수정은 `PATCH`를 올린다.
- 기능 단위 변경은 `MINOR`를 올린다.
- 호환성을 크게 깨는 변경이나 정식 공개 기준이 생기면 `MAJOR` 변경을 검토한다.
- 클라이언트 업데이트 감지는 `latest`와 `minimum` 버전을 기준으로 한다.

## 인증 상태

- 인증 화면은 명시적인 상태 모델을 따른다.
- 예: `checking`, `signed-out`, `profile-required`, `signed-in`, `reset-request`, `reset-update`.
- 로그인, 회원가입, 비밀번호 재설정, 프로필 생성 흐름을 한 상태에 섞지 않는다.

## 마이그레이션

- 마이그레이션 번호를 재사용하지 않는다.
- 이미 원격에 적용된 migration은 수정하지 않는다.
- 변경은 새 migration으로 추가한다.
- 원격 migration history와 로컬 파일 목록이 맞는지 확인한다.
- 기존 설정이나 데이터 구조를 크게 바꾸는 migration은 서버 점검 작업으로 취급한다.
- destructive migration은 별도 확인 후 진행한다.

## 서버 권위

- 클라이언트는 입력과 화면 표시를 담당한다.
- 전투 결과, 보상, 재화, 인벤토리 변경, 랭킹 기록은 서버가 확정한다.
- 서버 권위 로직은 필요한 시점에 Supabase Edge Functions로 추가한다.
